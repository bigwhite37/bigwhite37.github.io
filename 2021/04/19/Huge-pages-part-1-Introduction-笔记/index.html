为减少虚拟地址到物理地址的转换次数，虚拟内存缓存最近的转换，并称之为 Translation Lookaside Buffer (TLB)。TLB 可转换的所有内存称为 TLB reach。

使用 huge page 有两点好处：

  1. 地址转换减少，性能提升
  2. 地址转换信息一般存在 L2 cache 里，因而 application 在使用 huge page 是能用到 L2 cache 就更大，减少访问主存频率

大体上，使用 huge page 时，数据库性能能提升 2% - 7%，科学计算工作能提升 1% - 45%。

然而 huge page 并不是万能的，也因此 transpant huge page 的支持在主流操作系统中均有限。在一些 TLB 实现中，对 huge page 和 small page 所支持的 TLB entry 数量不同。如果 CPU 对 huge page 支持的 TLB entry 更少，则当访问模式非常分散、每个 huge page 引用数较少时，访问速度可能会较慢。另外 huge page 使用的虚拟地址空间的架构也会有限制。

2 Huge Page Fault Behaviour

接下来说明 huge page fault 的行为及 huge page 何时 allocate。此外，不同内核版本间，共享映射和私有映射也有重要的区别。

在 mmap() 调用时即会发生 huge page fault，这保证当 mmap() 成功返回时，所有对 shared mapping 的引用都会成功。而 private mapping 在 fork() 调用前都是安全的。当调用 fork() 时，子进程应尽早调用 exec() 或事先调用 madvise() 将 huge page mapping 标记为 MAD_DONOTFORK，否则 Copy-On-Write (COW) fault 可能会导致父进程或子进程因 allocation failure 事件而失败。

per-faulting pages 会急剧增加 mmap() 的开销，导致在 NUMA 上并非是最优的。自 2.6.18 起，第一次引用 huge page 时，其将像正常 mapping 一样 fault。为保证 fault 成功，会在 shared mapping 创建时即预置 huge page，相反，private mapping 则不做任何预置。这也意味着即使不调用 fork() 也可能失败。libhugetlbfs 在老内核上使用 readv() 处理 private mapping 的问题以保证该 mapping 的访问安全，不过该方式也不尽理想。

Pre-Faulting and Zeroing Optimizations

自 2.6.29 起，shared mapping 和 private mapping 都会有预置操作。且不论是什么进程访问 shared mapping 都将保证其 fault 成功。

对于 private mapping 而言，子进程数量是不确定的，因此仅保证对调用 mmap() 创建 mapping 的进程能成功 fault。当该进程（调用 mmap() 的进程）调用 fork() 时，会出现两个进程访问同一 page 的情况。当子进程发生 COW 时，会尝试 allocate 新 page。如果 allocate 成功，表示 fault 成功完成。如果 fault 失败，子进程终止，并在内核日志内指出 huge page 不足。如果是父进程上出现 COW，同样也会尝试 allocate 新 page。如果发生 allocation fail 事件，子进程页 unmap， 同时该事件将记录下来。如果父进程成功完成 fault 但子进程访问了 unmapped page，则子进程终止。

4 Huge Pages and NUMA

page fault 时，Linux 默认使用 node-local 策略分配内存。这一策略对 base page 和 huge page 都有效。

处理数据的线程应该是导致该数据 page fault 的线程。在 NUMA 上一种常见的反模式是当父线程初始化并设置好所有用到的内存，然后创建线程处理数据。在 NUMA 系统上，这可能导致一些工作线程访问相对于其所在 CPU 而言是远端的数据。在 NUMA 系统上，不论 page 大小均会出现该问题，而且访问同一 huge page 的工作线程在不同的 CPU 上，会出现更多的远端访问。

在 OpenMP 上使用 hug page 就有可能出现这一问题，其原因是 OpenMP 并不会按页边界划分数据。在使用 base page 时其实也会有问题，但使用 huge page 时由于会包含更多的数据，致使更有可能会发生该问题，从而导致 huge page 更有可能包含会不同线程处理的数据。当第一个线程创建 page 时，缺页中断使会在相对该线程所在 CPU 的本地内存载入数据。当这些数据不足以以 huge page 对齐的边界 split 时，该线程可能会在 fault 其所需的数据时连带着另一个线程将处理的数据（因为这部分数据范围在该 huge page 内）。之后，第二条线程会 fault 其剩余所需数据进本地内存，但仍会有部分数据需要去远端访问。该问题表现为在对同一工作运行次数越多，性能数据的标准差就越大。极端情况下，使用 huge page 的性能要低于使用 base page。因此，在 NUMA 上使用 huge page 时要慎重考虑数据的 split 边界。

上述例子的一个解决方案是结合使用 MPI 和 OpenMP。MPI 允许每个 NUMA 节点一个 MPI 进程，并将工作负载分配到这些 MPI 进程上。每个 MPI 进程都绑定都本地节点的 CPU 列表。节点内的并行化是使用 OpenMP 实现的，从而缓解了远程访问的问题。
